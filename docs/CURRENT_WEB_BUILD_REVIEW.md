# Shelter 2250 — 현재(Web 프로토타입) 제작 상태 리뷰 (재제작 관점)

## 1) 지금 “돌아가는 것”(자산으로 삼을 것)

- UI/레이아웃: 다중 캔버스 레이어 + 좌측 메뉴 + 우측 패널 구조가 있음(`index.html`, `css/style.css`)
- 코어 시스템(기본 골격):
  - 방 배치/제약/추천 위치(`js/shelter.js`, `js/room.js`, `js/game.js`)
  - 자원 생산/소비/패널티(`js/resources.js`, `js/game.js`)
  - 거주민 관리/배치 UI(`js/dweller.js`, `js/game.js`)
  - 저장/불러오기(슬롯 포함)(`js/game.js`)
- 문서/레퍼런스:
  - 세계관과 시스템 설계 문서가 이미 깊게 준비됨(`docs/**`)
  - UI 레퍼런스 이미지가 존재(`reference/**`)

## 2) 지금 “없는 것”(재미/완성도를 결정하는 구멍)

- 콘텐츠 루프가 미완:
  - 탐사 메뉴/이벤트 메뉴가 TODO 상태(`js/game.js`에 `coming soon`)
  - 경제가 “지출” 중심으로 흐르기 쉬움(캡 수입/지출처 설계가 더 필요)
- 튜토리얼/온보딩: 신규 유저가 무엇을 해야 하는지 안내가 부족
- 테스트/밸런스 자동화: “목표 지표”를 검증하는 루틴/로그 기준이 없음

## 3) 기술 부채(유지보수 리스크)

- 파일 비대화:
  - `js/game.js` 단일 파일이 매우 큼(약 1900줄) → 기능 추가 시 충돌/회귀 위험
  - 렌더/UI/시뮬이 한 클래스에 모이기 쉬운 구조
- 수치/룰의 하드코딩:
  - 방 비용/생산/소비/패널티가 코드 수정으로 움직임 → 밸런스 작업 비용 증가

## 4) 재제작 시 “버릴 것”과 “옮길 것”

### 버릴 것(그대로 이식 금지)
- Canvas 렌더링 구현 디테일(연출/그라데이션/픽셀 아트 등): Unity에서 다시 설계하는 편이 빠름
- UI 구조의 DOM/이벤트 결합 방식: Unity UI에 맞춰 재구성 필요

### 옮길 것(스펙/룰로서 가치 있음)
- 방 타입/자원 타입/배치 규칙/업그레이드 규칙의 “설계”
- 거주민 상태(HP/행복/허기/수분/방사능/특성)와 효율 계산의 “의도”
- 세계관 문서(지역/경제/타임라인/톤)와 UX 목표 문서

## 5) Unity 재제작을 위한 핵심 결론(총괄 지휘 관점)

1. 먼저 “시뮬레이션만 있는 Unity 버전”을 만든다(그래야 밸런스가 잡힌다).
2. 수치/정의는 ScriptableObject로 올리고, GameState는 JSON으로 저장한다.
3. 튜토리얼은 “첫 3일 생존”을 목표로 하는 5단계 가이드를 넣어야 한다.
4. 이벤트/탐사는 Phase 3 이후로 미루되, Phase 1에서 “붙일 자리(인터페이스)”는 만들어 둔다.
